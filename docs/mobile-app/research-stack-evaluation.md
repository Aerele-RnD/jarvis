# Research: Mobile Stack Evaluation — Jarvis B2B AI-Chat App on Frappe v16

> Supporting research for the Jarvis mobile app design (see `2026-07-06-mobile-backend-design.md`). Produced 2026-07-06 by a web-research agent; sources cited inline.

**TL;DR:** React Native + Expo (TypeScript) is the recommended stack, with Capacitor as a legitimate fast-track fallback. The single most decisive data point: Raven — the Frappe ecosystem's flagship chat app, with near-identical requirements (per-site OAuth login, socket.io chat, push relay, multi-site) — shipped its mobile v1 on Ionic + Capacitor and then **rewrote it on React Native + Expo** in 2025 ([Raven announcement](https://discuss.frappe.io/t/announcing-the-raven-mobile-app/145253), [Raven repo](https://github.com/The-Commit-Company/raven)). Flutter is ruled out for this team. Every Frappe-side capability needed (PKCE OAuth, token-auth socket.io, push subscribe API) exists in v15/v16, though the official push relay is not open source and Jarvis should plan to run its own.

---

## 1. Frappe OAuth2 provider for mobile

**Verdict: fully workable for a public mobile client. Authorization Code + PKCE + refresh tokens are all supported.**

- Frappe ships a built-in OAuth2/OpenID Connect provider (`frappe.integrations.oauth2`, based on oauthlib) with these endpoints ([official docs](https://docs.frappe.io/framework/user/en/guides/integration/rest_api/oauth-2), [source](https://github.com/frappe/frappe/blob/develop/frappe/integrations/oauth2.py)):
  - `GET /api/method/frappe.integrations.oauth2.authorize` (params: `client_id`, `state`, `response_type=code`, `scope`, `redirect_uri`, plus optional `code_challenge` / `code_challenge_method`)
  - `POST /api/method/frappe.integrations.oauth2.get_token` (grant `authorization_code` with `code_verifier`; also accepts `refresh_token` for refresh)
  - `revoke_token`, `introspect_token` (with `token_type_hint` of access/refresh token), and `openid_profile` (Bearer-token user info); ID tokens are OIDC JWTs with `aud/iss/sub/roles/exp/iat` claims.
- **PKCE**: both `s256` and `plain` challenge methods are supported; the implicit/token flow is deliberately **not** supported for security. PKCE landed via [PR #12997 (oauthlib refactor)](https://github.com/frappe/frappe/pull/12997) and is documented back to [v14 docs](https://docs.frappe.io/framework/v14/user/en/guides/integration/rest_api/oauth-2) — present and stable in v15 and v16. (Frappe v16 went stable Dec 2025 with no breaking OAuth changes reported: [release plan thread](https://discuss.frappe.io/t/erpnext-hrms-frappe-framework-v16-release-dates/156349).)
- **OAuth Client doctype**: redirect URIs as a space-separated list with a default; "Skip Authorization" can suppress the consent screen for first-party apps ([setup guide](https://docs.frappe.io/framework/user/en/guides/integration/how_to_set_up_oauth)). Custom-scheme redirects (e.g. `jarvis://auth`) work — this is exactly how existing Frappe mobile apps do it.
- **How existing Frappe mobile apps authenticate**: Raven's mobile app has users generate an OAuth Client from Raven's web settings ("Settings → Mobile App") and logs into any per-customer site via that client ([forum thread, posts #7–8](https://discuss.frappe.io/t/announcing-the-raven-mobile-app/145253/7)); it requires Frappe v15.63+ — a hint that some OAuth/notification fixes landed mid-v15, so pin the minimum Frappe version accordingly. An older precedent: [revant's AndroidFrappeAuthenticator](https://github.com/revant/AndroidFrappeAuthenticator) (native Android account-manager OAuth for Frappe).
- **Multi-tenant wrinkle (Jarvis-specific)**: since every customer runs their own site, the app must do "identify your site → discover/create OAuth client → PKCE flow". Raven's UX (each site pre-provisions an OAuth Client; the mobile app asks for site URL) is the pattern to copy. Since Jarvis controls site provisioning in jarvis_admin, it can auto-create the OAuth Client fixture on every tenant site — smoother than Raven's manual step.

| "Hussh" | **No public evidence found** of a Frappe app by this name — treat as nonexistent. | — |
| Frappe push relay | `frappe.push_notification.subscribe/unsubscribe?fcm_token=` endpoints + `FrappePushNotification` JS client, config served by `push-notification-relay.frappe.cloud` (`notification_relay.api.get_config`), FCM-based ([Frappe wiki client reference](https://github.com/frappe/frappe/wiki/Frappe-Push-Notification-Client-Reference)) | **The relay server itself is not open source** ([hrms #4470](https://github.com/frappe/hrms/issues/4470) requests it). Community self-hosted alternatives: [erpnext_fcm](https://github.com/tridz-dev/erpnext_fcm), [Frappe-Push](https://github.com/maxfu9/Frappe-Push). **Takeaway:** Frappe's own trajectory is PWA (Vue/Ionic) for simple employee apps and React Native + Expo for the serious chat app; Flutter was tried and abandoned. Everyone hits the same push-relay gap on self-hosted sites, and the emerging pattern is "small vendor-run relay + tokens stored per user/device on each site."

## 3. Stack comparison for this team (Python / TypeScript / Vue 3, no mobile devs)

### React Native + Expo
- **OAuth**: [expo-auth-session](https://docs.expo.dev/versions/latest/sdk/auth-session/) does Authorization Code + PKCE against any compliant provider (proven against Keycloak etc. — [worked example](https://rene-wilby.de/en/blog/rn-expo-oauth-authorization-code-flow-pkce-keycloak/)); Frappe's PKCE support means no client secret in the app.
- **Push**: [expo-notifications](https://docs.expo.dev/versions/latest/sdk/notifications/) covers FCM + APNs, and supports **notification categories with action buttons** (`setNotificationCategoryAsync`) — i.e. Approve/Deny buttons on the notification, mapping onto the ERP write-confirmation flow ([docs](https://docs.expo.dev/versions/latest/sdk/notifications/), [guide](https://docs.expo.dev/push-notifications/what-you-need-to-know/)). *(Platform caveat found in later verification: action handlers only run reliably for foregrounded/backgrounded-alive apps — killed-app actions must deep-link into the app. See the backend design changelog.)* Expo's push service is optional: expo-notifications works with native device tokens and direct FCM/APNs ([docs](https://docs.expo.dev/push-notifications/sending-notifications-custom/)); Python side can use [firebase-admin](https://firebase.google.com/docs/cloud-messaging/auth-server) or [exponent_server_sdk](https://github.com/expo-community/expo-server-sdk-python).
- **socket.io**: socket.io-client works in RN; the scary GitHub issues ([#1245](https://github.com/socketio/socket.io-client/issues/1245), [#1350](https://github.com/socketio/socket.io-client/issues/1350)) are from the v2 era (2018–2020); v4 is routinely used in RN chat apps — Raven's realtime runs on it. Force `transports: ['websocket']` and pass the Frappe token via `extraHeaders`/`auth`.
- **Secure storage**: [expo-secure-store](https://docs.expo.dev/versions/latest/sdk/securestore/) = iOS Keychain / Android Keystore-encrypted prefs; keep values under ~2KB per key — fine for tokens.
- **Voice**: [expo-speech-recognition](https://github.com/jamsch/expo-speech-recognition) wraps SFSpeechRecognizer/Android SpeechRecognizer with `iosTaskHint: 'dictation'`, on-device options, actively maintained; [@react-native-voice/voice](https://github.com/react-native-voice/voice) is the older alternative.
- **OTA**: [EAS Update](https://www.ecloudbuzz.com/best-ota-update-solutions-for-mobile-apps/) is the most polished OTA story in mobile; JS-bundle updates are Apple-policy-compliant for fixes/tweaks ([Bitrise policy explainer](https://bitrise.io/blog/post/what-app-stores-allow-with-ota-updates-apple-and-google-policy-explained)).
- **Team fit**: TypeScript carries over 100%; React is new but Vue 3 Composition API devs adapt fast (hooks ≈ composables). JS/TS talent pool dwarfs Dart (~20:1; RN roles fill in 3–4 weeks vs 6–8 for Flutter — [comparison data](https://techsy.io/en/blog/react-native-vs-flutter), [dev.to salary/job comparison](https://dev.to/arshtechpro/flutter-vs-react-native-vs-native-2025-which-is-better-salary-job-comparison-3bpc)). New Architecture (Fabric/JSI) gives near-native perf for chat lists ([2026 comparison](https://thedebuggersitsolutions.com/blog/cross-platform-app-2026-flutter-react-native-capacitor)).
- **Cost**: Vue components can't be reused; the chat UI is rewritten in React (but API clients, types, message models can live in a shared TS monorepo package, as Raven does with ~40% sharing).

### Flutter
- All capabilities exist and are mature: [flutter_appauth](https://pub.dev/packages/flutter_appauth) (AppAuth wrapper, PKCE-native), firebase_messaging, flutter_secure_storage, [speech_to_text](https://pub.dev/packages/speech_to_text) (mature but explicitly aimed at "commands and short phrases", not long dictation), socket_io_client Dart port (version-matching quirks with server versions). OTA exists only via [Shorebird](https://docs.shorebird.dev/code-push/) code push (a modified Flutter engine; Dart patches, restartless supported — [2026 guide](https://dev.to/techwithsam/how-to-push-over-the-air-ota-flutter-updates-with-shorebird-complete-2026-guide-4d35)).
- Performance is best-in-class (Impeller, AOT Dart — [2026 analysis](https://www.bolderapps.com/blog-posts/flutter-vs-react-native-in-2026-why-the-new-architecture-and-impeller-2-0-changed-everything)), but for a chat UI RN's New Architecture is entirely sufficient.
- **Disqualifiers for this team**: zero skill overlap (Dart, widget paradigm), zero code/type sharing with the Vue/TS web app and Python backend, hardest hiring pool, and the Frappe ecosystem's own Flutter effort ([frappe/mobile](https://github.com/frappe/mobile)) is abandoned — no ecosystem tailwind.

### Capacitor (wrap the existing Vue 3 + frappe-ui SPA)
- **Fastest possible start**: ship the existing chat UI in a native shell; Frappe HR's Vue+Ionic PWA proves the general pattern in this ecosystem.
- **OAuth**: [@byteowls/capacitor-oauth2](https://www.npmjs.com/package/@byteowls/capacitor-oauth2) / [capacitor-community/generic-oauth2](https://github.com/capacitor-community/generic-oauth2) support code flow **only with PKCE** (they refuse secret-based flows on-device); Capawesome also ships OIDC+PKCE and Keychain/Keystore secure-storage plugins ([Capawesome plugin suite](https://capawesome.io/sdks/capacitor/)).
- **Push**: [@capacitor/push-notifications](https://capacitorjs.com/docs/apis/push-notifications) handles FCM/APNs tokens and taps, **but action buttons are not in the core plugin** — a years-open feature request ([capacitor-plugins #213](https://github.com/ionic-team/capacitor-plugins/issues/213)); workarounds are the local-notifications plugin or a community fork ([HaylLtd/capacitor-push-notifications](https://github.com/HaylLtd/capacitor-push-notifications)).
- **Voice**: [@capacitor-community/speech-recognition](https://github.com/capacitor-community/speech-recognition) is actively maintained (v7, 2025 releases).
- **Background/streaming weaknesses (the big ones for Jarvis)**: iOS suspends the WKWebView the moment the app backgrounds; WebKit has killed webview WebSocket instances on suspension ([WebKit bug 228296](https://bugs.webkit.org/show_bug.cgi?id=228296), [Apple dev thread](https://developer.apple.com/forums/thread/685403)); the webview process can even be terminated under memory pressure, breaking push handler wiring ([capacitor #6676](https://github.com/ionic-team/capacitor/issues/6676)). Reliable push+socket behavior ends up requiring native plugin work anyway ([Signal-style Capacitor guide](https://lushbinary.com/blog/android-ios-push-notification-websocket-capacitor-signal-guide-2026/), [Background Runner](https://capacitorjs.com/docs/v5/apis/background-runner)).
- **App Store risk**: Guideline 4.2 "minimum functionality" rejections target webview-wrapper apps; mitigation is exactly the native surface you'd have to add (push, native navigation, offline) ([MobiLoud guideline analysis](https://www.mobiloud.com/blog/app-store-review-guidelines-webview-wrapper), [Ionic forum rejection report](https://forum.ionicframework.com/t/app-store-rejection-4-2-design-minimum-functionality-my-first-after-2-years-of-ionic/200908)). A polished chat app with push and voice would likely pass, but it's review-roulette a B2B product doesn't want.
- **OTA**: excellent — [Capgo](https://capgo.app/) is open-source and **self-hostable** ([self-hosted docs](https://capgo.app/blog/self-hosted-capgo/)).
- **Perf**: webview chat streaming with heavy lists/animations is the known weak spot vs RN/Flutter ([Kanopy comparison](https://kanopylabs.com/blog/capacitor-vs-react-native-vs-flutter)) — and Raven's team lived this and migrated off it.

## 4. Push architecture

- **Server → device**: FCM HTTP v1 + APNs. From Python, [firebase-admin](https://excellencetechnologies.in/blog/firebase-push-notification-with-python/) is the recommended SDK (service-account JSON, no legacy API keys); [pyfcm](https://pypi.org/project/pyfcm/) still works but firebase-admin has effectively superseded it (see [fcm-django docs](https://fcm-django.readthedocs.io/en/latest/)). Expo's push service ([Python expo-server-sdk](https://github.com/expo-community/expo-server-sdk-python)) is the simplest alternative but adds a third-party hop (data-locality customers may object).
- **Multi-tenant reality**: APNs/FCM credentials are bound to *the vendor's* app bundle ID, so per-customer Firebase projects are impossible with one store app. This forces the architecture Frappe and Raven both converged on: **each tenant site stores device tokens per user/device** (Frappe core already exposes `frappe.push_notification.subscribe/unsubscribe` — [wiki reference](https://github.com/frappe/frappe/wiki/Frappe-Push-Notification-Client-Reference)) and **relays sends through one central, vendor-run push gateway** holding the FCM/APNs credentials. Frappe Cloud's relay (`push-notification-relay.frappe.cloud`) is exactly this but closed-source ([hrms #4470](https://github.com/frappe/hrms/issues/4470)); Raven is building an anonymized token-only relay ([raven #1318](https://github.com/The-Commit-Company/raven/issues/1318)). For Jarvis: build a small relay service (fits the existing fleet-agent/gateway pattern); tenant sites call it with the push envelope; store only tokens/counters, no user PII.
- **Data-locality escape hatch**: for customers refusing Google entirely, [ntfy](https://docs.ntfy.sh/faq/) (self-hostable, UnifiedPush) or [Gotify](https://tuta.com/blog/google-push-alternative) are credible self-hosted options — but they can't reach a *store-distributed iOS app* (APNs is unavoidable on iOS). Realistic positioning: FCM/APNs with anonymized relay by default; ntfy only for special Android-fleet deployments.
- The 5–12-minute agent tasks are the textbook push use case: the app **will** be suspended by then on iOS; deliver an alert push with an approval category, deep-link into the conversation, and refetch state on open.

## 5. Voice input

- **On-device dictation (recommended baseline)**: iOS SFSpeechRecognizer / Android SpeechRecognizer — free, fast, private. Plugins: [expo-speech-recognition](https://github.com/jamsch/expo-speech-recognition) (RN; on-device mode, dictation task hint, actively maintained), [@react-native-voice/voice](https://github.com/react-native-voice/voice), Flutter [speech_to_text](https://pub.dev/packages/speech_to_text) (mature, short-phrase oriented; Android auto-stops on silence), [@capacitor-community/speech-recognition](https://github.com/capacitor-community/speech-recognition) (active, v7 2025).
- **Server transcription (upgrade path)**: OpenAI whisper-1 and gpt-4o-transcribe at ~$0.006/min, gpt-4o-mini-transcribe ~$0.003/min; gpt-4o-transcribe cuts WER ~22% vs whisper-v3 ([2026 pricing roundup](https://diyai.io/ai-tools/speech-to-text/openai-whisper-api-pricing-2026/)); Deepgram Nova-3 ~$0.0043/min with <300ms streaming ([Gladia comparison](https://www.gladia.io/blog/best-whisper-alternatives-2026)); self-hosted [faster-whisper/whisper.cpp](https://brasstranscripts.com/blog/openai-whisper-api-pricing-2025-self-hosted-vs-managed) for data-locality customers — slots naturally behind the existing LLM gateway. Recommended: on-device dictation for v1.1 (zero cost, zero data egress), record-and-upload to Whisper-class models as a settings-gated enhancement.

## 6. Streaming resilience on mobile

- **Frappe realtime works with mobile token auth**: the socket.io server (Node, `/realtime`, Redis pub-sub from Python) authenticates via SID cookie **or an Authorization header — explicitly documented for mobile apps** — and namespaces per site as `/{sitename}` ([Frappe realtime docs](https://docs.frappe.io/framework/user/en/api/realtime)), which fits the one-app-many-sites model directly.
- **Reconnect**: socket.io auto-reconnects with backoff, but has **no built-in event replay** — events emitted while disconnected are simply lost; SSE's `Last-Event-ID` resume is the protocol-level fix ([WebSocket.org SSE comparison](https://websocket.org/comparisons/sse/), [Ably analysis](https://ably.com/blog/websockets-vs-sse)) but Frappe doesn't ship SSE — don't fight the framework.
- **The pattern that actually works** (and what chat vendors converge on): treat the socket as a *foreground-only accelerator* and make the database the source of truth. Jarvis messages are already persisted in doctypes, so: (1) on connect/reconnect/foreground, fetch messages past a cursor; (2) render live tokens/updates from socket events while foregrounded; (3) rely on push for anything that completes while backgrounded. This is mandatory anyway because iOS terminates apps holding open WebSockets in background ([react-native #26731](https://github.com/facebook/react-native/issues/26731), [WebKit 228296](https://bugs.webkit.org/show_bug.cgi?id=228296)).
- For token-level streaming of in-progress LLM output, chunked socket.io events + periodic server-side checkpointing of partial output (so a reconnect refetch shows the partial message) is the resilient design ([RxDB realtime protocol survey](https://rxdb.info/articles/websockets-sse-polling-webrtc-webtransport.html)).

## Final recommendation

**Build React Native + Expo (TypeScript), and copy Raven's architecture.** Reasoning for *this* team:

1. **The requirements are native-heavy, not UI-heavy.** Push with approval actions, voice dictation, background-survival semantics, and secure OAuth token storage are exactly where Capacitor needs forks and native plugins ([action buttons gap](https://github.com/ionic-team/capacitor-plugins/issues/213), [webview socket suspension](https://bugs.webkit.org/show_bug.cgi?id=228296)) and where Expo has first-party, documented modules.
2. **The ecosystem already ran this experiment.** Raven = Frappe chat + OAuth + socket.io + push relay; it launched on Capacitor and rewrote on Expo for reliability of notifications and interaction smoothness ([announcement](https://discuss.frappe.io/t/announcing-the-raven-mobile-app/145253)). Paying the Capacitor→RN rewrite tax later is the predictable failure mode; Raven's open codebase is also a free reference implementation for the Frappe-OAuth-PKCE flow, realtime auth, and relay design.
3. **TypeScript preserves most of the team's leverage.** API clients, message schemas, and streaming state machines can live in a shared TS package consumed by both the Vue web app and the RN app (Raven shares ~40% this way). Only the view layer is rewritten — and hiring for React/TS is the easiest in mobile ([job-market data](https://techsy.io/en/blog/react-native-vs-flutter)).
4. **Flutter is eliminated**: no skill overlap, no code sharing with either the Vue frontend or Python backend, thinnest hiring pool, and Frappe's own Flutter app is abandoned ([frappe/mobile](https://github.com/frappe/mobile)).
5. **Keep Capacitor as the explicit fallback**, not a compromise default: if a 4–6 week "app in stores now" deadline appears, wrapping the Vue SPA with push + speech plugins is viable (Frappe HR PWA precedent), accepting the 4.2 review risk and the known ceilings — but scope it as a stopgap.

**Proposed v1 architecture:** Expo app (expo-auth-session PKCE against `frappe.integrations.oauth2` with a per-tenant auto-provisioned OAuth Client; tokens in expo-secure-store; refresh via `get_token` with `refresh_token`) → socket.io-client with Authorization header, foreground-only, cursor-based catch-up fetch on reconnect/foreground → self-built minimal push relay (Python, firebase-admin; tokens registered per user/device on each tenant site; anonymized token-only relay) → expo-notifications with an approval category → expo-speech-recognition for dictation (v1.1), server-side Whisper-class transcription as an optional upgrade behind the LLM gateway → EAS Update for OTA fixes.
