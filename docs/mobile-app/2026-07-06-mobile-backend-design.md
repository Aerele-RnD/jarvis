## What the adversarial pass changed (changelog)

The first synthesis assumed notification action buttons could commit approvals from a killed/backgrounded app. Platform verification killed that: v1 notification actions now **deep-link into an in-app biometric approval sheet** (in-place background actions deferred to v1.1+ as a native-module enhancement). Security verification added: a **device-registry whitelist** in the auth hook (because Frappe never revokes superseded refresh tokens), a **method allowlist** for mobile bearer tokens (scope-`all` containment), a **template-based relay API** (leaked relay key can no longer send free-form lock-screen content), **conversation-bound `stop_run`** (cross-user cancel closed), disabling Frappe's **guest dynamic OAuth client registration** (ships enabled by default), and **2-digit number-matching** in QR pairing. Frappe-source verification added the missing `active_run_id` plumbing (no run id is persisted today) and corrected secure-store/silent-push assumptions.

## Decisions & rationale

1. **Base = the security-lens design** (both judges independently). Its pairing flow, revocation model, and fail-closed posture survive adversarial review; its extra machinery is threat-traceable, not gratuitous.
2. **Two-step pairing over a single redeem-URL.** Passing the raw pairing token as a GET param to a guest endpoint lands it in phone-browser history and nginx access logs (the classic secrets-in-logs finding). The claim→browser_code exchange closes this for one extra round-trip.
3. **Consent screen stays ON (`skip_authorization=0`).** Honest framing: it is *not* the primary QR-hijack control (an attacker self-approves); TTL/one-time/hashing/web-visibility are. But it costs one tap, covers the non-QR fallback login path, and shows the named client. Note: consent is contingent on a *site-level* toggle — see §6 (pin + monitor).
4. **Pairing live-state in Redis only; slim append-only audit-log doctype kept.** Redis gives native TTL + atomic single-use with zero at-rest secrets; the log row (no secret material) gives "device paired — not you? revoke" visibility and forensics.
5. **Relay = standalone `jarvis-push-relay` service, not a `jarvis_admin` module.** An admin-module relay couples every tenant's push latency to admin-bench health, runs APNs HTTP/2 connection management inside gunicorn request workers, and parks Apple/Google signing creds on the plane carrying the worst open audit findings (SEC-J, SSRF — audit §9). Kept from the minimal lens anyway: the stateless zero-retention contract and key distribution over the existing `get_connection` channel. Kept from the ops lens: tenant lifecycle wiring and delivery observability.
6. **`stop_run` ships in v1.** A user watching a runaway turn on a phone (worker budget 720 s, `jarvis/chat/api.py:37`) needs recourse. Scoped as a cooperative Redis flag — no container kill, no worker-interrupt plumbing.
7. **Catch-up = seq-cursor `sync_conversation` + `home_snapshot`.** A `(modified, name)` cursor was rejected: same-second ties and row mutation can skip/replay rows under concurrent commits; `seq` already exists and is monotonic per conversation (`jarvis/chat/api.py:821-827`).
8. **Push privacy defaults:** content previews default **OFF**; no confirm/approval tokens ever ride push payloads. Per-user two-class notification toggles ship in v1; quiet hours defer to v1.1.
9. **IDOR stance = hard gate:** the audit's JRV-SEC-002 + SEC-B owner-check fixes are a **mobile-GA prerequisite**, not a "recommended piggyback" (bearer tokens widen those endpoints' exposure).
10. **No plaintext access token in the device row.** Frappe's `OAuth Bearer Token` row name **is** the access-token string (`frappe/oauth.py:107-111` looks tokens up by `name`), so storing the "token name" would persist a live credential — the exact JRV-SEC-005 pattern this design avoids. The device row stores `token_hash = sha256(access_token)` only.
11. **(Post-verification) Approve/deny is an in-app moment, not a background action.** Expo's notification-response listeners never fire for non-foreground actions when the app is killed; Android will not render category buttons on raw-FCM data messages for a killed app (expo#36282); iOS background JS handlers drop async fetches (expo#14004); and biometric prompts are impossible from a background handler. v1: every notification action deep-links into a focused approval sheet (foreground, biometric, then commit). Native in-place actions (iOS `didReceiveNotificationResponse`, Android notifee + data messages) are a v1.1+ progressive enhancement.
12. **(Post-verification) Revocation = device-registry whitelist, not blacklist.** Frappe never invalidates superseded refresh tokens on rotation (`frappe/oauth.py:282-292` filters `status: Active` only), so tracking one token hash per device cannot kill a stolen refresh chain. Inverted: for the mobile OAuth client, **only bearer tokens whose hash matches an active device row are accepted at all** (§1).

## 1. Auth & pairing

**OAuth Client auto-provisioning.** New `jarvis/mobile/oauth_client.py::ensure_mobile_oauth_client()`, appended to the existing `after_migrate` list (`jarvis/hooks.py:164-169`). Idempotently creates one **public** `OAuth Client` per tenant site: `app_name="Jarvis Mobile"`, `grant_type="Authorization Code"`, `response_type="Code"`, `token_endpoint_auth_method="None"` (public client — `oauth_client.py:71-72 is_public_client()`; advertised in server metadata `frappe/integrations/oauth2.py:320`), `redirect_uris="jarvis://auth"`, `scopes="all openid"`, `skip_authorization=0`, allowed_roles defaulting to System User (`oauth_client.py:62-64`). PKCE S256 is validated when a challenge is present (`frappe/oauth.py:155-162`), and a code minted with a challenge but redeemed without a verifier is **deleted, not just refused** (`frappe/oauth.py:149-153`). The refresh grant works for public clients: `authenticate_client` loads by `client_id`/refresh token with no secret check (`frappe/oauth.py:94-119`) and the grant allowlist includes `refresh_token` (`frappe/oauth.py:182-185`). `client_id` is public; the app fetches it via `app_config` (§3) or the QR.

**Accepted limitation — PKCE is not server-mandated.** Frappe enforces PKCE only when the authorization request *includes* a `code_challenge`; a challenge-less flow against the same public client is technically possible. Compensations: authorization codes are single-use and short-lived, the official app always sends S256, and the consent screen names the client. Recommended follow-up: a small upstream patch (or server-side override) requiring `code_challenge` for clients with `token_endpoint_auth_method="None"`.

**Same-PR hardening (post-verification):** Frappe ships **guest dynamic OAuth client registration enabled by default** (`oauth_settings.json` default 1; endpoint at `frappe/integrations/oauth2.py:335-344`). The mobile PR disables it on every tenant site (settings patch) and adds a bench health-check that alerts if `skip_authorization` or dynamic registration drift from the pinned values.

**QR pairing flow (end-to-end).**
1. Logged-in web SPA → `jarvis.mobile.pairing.create_pairing_session()` (cookie+CSRF). Server mints `pid` + 256-bit `secret`, stores **only `sha256(secret)`** in Redis (`jarvis:mpair:<pid>`, TTL 120 s, status `pending`, bound to `frappe.session.user` + hash of the web `sid`), inserts a `Jarvis Mobile Pairing Log` row (no secrets). Returns `{pid, secret, site_url, client_id, expires_at}`. Web renders QR `{v:1, site, pid, secret, client_id}`, polls `pairing.status(pid)`, auto-rotates the QR at 90 s.
2. App scans, optionally validates the site via `/.well-known/openid-configuration` (`oauth2.py:269-273`), then POSTs `pairing.claim(pid, secret)` (guest, rate-limited, fail-closed). Server verifies the hash, atomically flips `pending→claimed` (GETDEL semantics, mirroring `jarvis/chat/pending_confirm.py:141-142`), and returns a fresh single-use **60 s `browser_code`** plus a **2-digit match code**. The QR secret travels only in a POST body — never a URL.
3. **Number-matching (post-verification):** the app displays the 2-digit code; the web page (via `pairing.status`) asks "Does your phone show **47**? [Yes, continue] [No — cancel]". This closes the shoulder-surf-hijack and reverse-phishing (attacker QR pairing a victim's phone onto the attacker's account) paths that TTL/one-time alone do not. Web confirmation flips the session to `confirmed`; only then does `web_login` accept the browser code.
4. App generates its PKCE verifier/challenge and opens the **system browser** (expo-auth-session) at `pairing.web_login?pid=…&code=…&redirect-to=<authorize URL>`. `web_login` consumes `browser_code` (single-use, GETDEL), creates a Frappe session for the paired user (flagged `mobile_pairing_temp`, 10-min expiry), and 302s → `frappe.integrations.oauth2.authorize` (`oauth2.py:92-93`) — without this session the guest branch would force password entry on the phone (`oauth2.py:97-100`). **`redirect-to` is allowlisted**: it must be this site's own `authorize` endpoint with this site's mobile `client_id` — anything else is rejected (open-redirect from a freshly-minted session otherwise).
5. User sees the consent screen ("Jarvis Mobile", `oauth2.py:126-138`), taps Allow → `approve` (`oauth2.py:66`) redirects `jarvis://auth?code=…`. App exchanges code+verifier at `get_token` (`oauth2.py:143-144`) → `{access_token, refresh_token, expires_in}`. Pairing session → `completed`; web poll shows "Phone paired ✓ — Not you? Revoke".
6. App calls `register_device` (§2); server deletes the temp browser session and closes the pairing log entry.

**Fallback (no QR):** type the site URL in-app → same system-browser authorize URL → normal Frappe login → consent → PKCE exchange. Pairing session and number-matching not involved.

**Token lifecycle.** Access token TTL 3600 s — the oauth server is built with no expiry override (`oauth2.py:37-42`), and `expires_in` is persisted verbatim (`frappe/oauth.py:210-213`); every API request validates expiry + revocation + client-level scopes (`frappe/auth.py:647-684`; `frappe/oauth.py:227-235`). Refresh tokens rotate on each refresh grant (new `OAuth Bearer Token` row via `save_bearer_token`). **Client-side rules (post-verification, platform-mandated):** tokens live in expo-secure-store with `keychainAccessible = AFTER_FIRST_UNLOCK` (the default `WHEN_UNLOCKED` makes the token unreadable in exactly the locked-phone scenarios push handling hits); the app **never refreshes tokens from a background handler** (an interrupted rotation permanently bricks the chain) — refresh happens in foreground only, proactively when <5 min remain, and re-calls `register_device` (updates `token_hash`). A 401 in a background context simply defers to next foreground. Refresh-failure responses from Frappe are not always OAuth-shaped JSON — the client treats any non-2xx refresh as "re-auth needed" rather than parsing error bodies.

**Per-device revocation — device-registry whitelist (post-verification).** Frappe does not bind tokens to devices, and superseded refresh tokens stay `Active` forever (`frappe/oauth.py:282-292`), so a stolen refresh token would outlive any single-hash blacklist. Design inverted to a whitelist: a request-time hook `jarvis/mobile/auth.py::enforce_mobile_token_policy`, registered via `auth_hooks` (executed for every request — `frappe/auth.py:639,742-744`), applies to bearer sessions belonging to the mobile OAuth client and **401s any token whose `sha256` does not match an enabled, non-revoked `Jarvis Mobile Device` row**. Legit apps re-register after every refresh, so their current token is always whitelisted; tokens minted from a stolen stale refresh chain are unregistered → dead on arrival for API use. If a thief instead registers their own device row (they hold a briefly-valid bearer), that creates a **new device** → fires a "new device signed in" notification to all of the user's devices and the web UI (visibility instead of silence). *Sign out this device* = revoke the matched `OAuth Bearer Token` row (framework write, `frappe/oauth.py:247-263` — also kills its refresh token), set `revoked=1`, blank the push token. *Sign out everywhere* = revoke **all** Active tokens for user×mobile-client. Residual risk (stolen refresh + attacker-registered device) is bounded by the new-device alert + everywhere-signout, and documented.

**Where secrets live.** Phone: access+refresh tokens in expo-secure-store (`AFTER_FIRST_UNLOCK`) only; app binary carries nothing (public client). Bench: pairing secrets hashed-in-Redis; device rows hold token *hashes*; relay key in `Jarvis Settings.push_relay_key` (Password, written via `set_encrypted_password` — never `db_set`, JRV-SEC-005, audit §14). Relay: only APNs p8 + FCM service-account files on disk.

## 2. Push pipeline

**Device registration (tenant site), `jarvis/mobile/api.py`:**
- `register_device(device_uuid, platform, push_token, push_provider, device_name, app_version, os_version)` — bearer-auth; upserts `Jarvis Mobile Device` keyed `(user, device_uuid)`; sets `token_hash` from the request's own Authorization header; doubles as heartbeat (`last_seen_at`). Owner-scoped by explicit `user = frappe.session.user` SQL — never `if_owner`/bare `get_doc` (the JRV-SEC-002 trap, audit §14). Creating a **new** `(user, device_uuid)` row triggers the "new device signed in" notification (§1).
- `unregister_device(device_uuid)` (sign-out), `list_devices()`, `revoke_device(device_uuid)` (lost phone; callable from a web session too).

**Central relay — `jarvis-push-relay`.** New repo, FastAPI/uvicorn in the fleet-agent idiom (TOML config, pydantic wire models); a fifth vendor-plane box co-located with the admin host, bound `127.0.0.1` behind nginx TLS, dedicated service user — explicitly the corrected posture, not fleet-agent's shipped root/`0.0.0.0` (SEC-E/D9, audit §9 + §14). Benches call it; it never calls benches (one-way trust).

- **API (template-based — post-verification):** `POST /v1/send` `{messages:[{token, platform, category, template, vars, badge, collapse_id, priority}]}` → per-message `{ok|unregistered|throttled|error}`. **The relay holds the notification text templates** (per category, length-capped variable slots); tenants supply only `template` + `vars` — **no free-form title/body crosses the relay API**. A leaked tenant key can therefore only emit well-formed, Jarvis-shaped notifications at capped rates to tokens the attacker already possesses — not arbitrary lock-screen content. Admin surface: `POST /v1/tenants`, `DELETE /v1/tenants/{id}`, `POST /v1/tenants/{id}/rotate-key` (separate admin bearer); `GET /healthz` (unauth, no version string), `GET /metrics`.
- **Platform mapping (post-verification):** iOS → alert push with `category`, `interruption-level: time-sensitive` for the approval class (requires the Time-Sensitive entitlement), and a server-computed **`badge`** (pending-approvals count — the app cannot update its own badge while killed). Android → **data message + client-side notifee render** (raw-FCM notification messages cannot carry action categories reliably to a killed app), routed to one of two channels: `jarvis_tasks` (default priority) / `jarvis_decisions` (high priority).
- **Tenant→relay auth:** per-tenant bearer key, stored **hashed** relay-side, constant-time compare. Admin app is registrar: at customer activation `jarvis_admin` registers the tenant and the minted key rides the **existing authenticated `get_connection` payload** (admin→tenant contract, audit §3) into `Jarvis Settings.push_relay_url` + `push_relay_key`.
- **Abuse controls:** per-tenant rate limit (60/min sustained, burst 120) **fail-closed**; 4 KB payload cap; template/category allowlist; volume-anomaly alerting; admin key rotation. True tenant↔token binding is impossible with zero retention — accepted, with the template + quota + rotation mitigations above.
- **Data at rest:** tenant registry (id, key-hash, counters, disabled flag) + a 7-day delivery log of `(tenant, token-hash-prefix, category, status code, latency)`. **No device tokens at rest, no payload content, ever** — the relay is a mere conduit (DPA: processor with no content access).

**Event sources.** One helper `jarvis/mobile/push.py::notify_user(user, kind, payload)`, always enqueued on the `short` queue (never blocks a turn), invoked at the existing emit sites:

| Kind | Hook point |
|---|---|
| `run:end` | beside `jarvis/chat/turn_handler.py:780-781` |
| `run:error` | `turn_handler.py:512-513, 756-757, 1187-1188`; stale-scan `jarvis/chat/stale_scan.py:72-73, 184-185` |
| `action:pending` (write parked) | beside `jarvis/api.py:682-690` |
| `approval:pending` | **new** `doc_events` `after_insert` on `Jarvis Approval Request` (added to `jarvis/hooks.py:244-247`; approval creation currently emits nothing — `approvals_api.py` only reads/decides) |
| `approval:decided` | after the flip in `approvals_api.decide` (`jarvis/chat/approvals_api.py:308-318`) — see delivery note below |
| proactive / macro done | `jarvis/chat/proactive.py:58-61`; `jarvis/chat/macros.py:356` |

Sent unconditionally; the foregrounded app suppresses locally (no server-side presence detection — non-goal).

**Cross-device cleanup (post-verification).** The originally-designed *silent* `approval:decided` retraction push is not implementable: silent pushes cannot replace a visible notification, iOS budgets them (~2-3/hour), and force-quit apps never receive them. v1 behavior: when an approval is decided, other devices receive a **visible, default-priority replacement** via the same `collapse_id` ("Approval handled ✓ — Sales Invoice INV-00042"), which overwrites the still-pending alert where the OS supports it; everywhere else, in-app state + the server-computed badge reconcile on next foreground.

**Payload + deep links.** Data-minimal default: template-rendered generic text ("Jarvis finished a task", "Approval needed") + `data:{v:1, kind, conversation, run_id, approval, deep_link}`; `include_content_in_push=1` (per-user opt-in, default **0**) adds a 120-char snippet slot. Deep links: `jarvis://conversation/<id>`, `jarvis://approval/<name>`, `jarvis://pending/<conversation>`. `collapse_id = run_id|approval`.

**Approve/Deny round-trip (post-verification — foreground commit).** **No confirmation/approval tokens in push payloads** — they'd transit Google/Apple and the relay. Notification actions ("Approve…", "Deny…", "View") and plain taps all **deep-link into the app**: the app opens directly onto a focused approval sheet, prompts biometric (expo-local-authentication), then calls the existing endpoints — `approvals_api.decide(name, decision, approve)` (`approvals_api.py:289-357`; the conditional `Pending→decided` UPDATE makes double-submission race-safe `:308-318` and resumes the conversation via `send_message` `:353`) or, for write-confirmations, `list_pending_confirmations(conversation)` (`actions_api.py:316-347`) then `confirm_tool(token, conversation)` (`actions_api.py:235-313` — owner-bound, conversation-bound, single-use, 900 s TTL: `pending_confirm.py:28`, consume semantics `:103-126`). Expired/decided → the sheet shows "Already handled" state. On iOS, action buttons appear on long-press/expand; a plain tap lands on the same sheet — the UX target is lock screen → decided in under 5 seconds either way. In-place background actions (no app open) are v1.1+: iOS `didReceiveNotificationResponse` in native code, Android notifee action handlers — explicitly out of v1.

**Delivery failure handling.** Relay: 3 retries with backoff on transient FCM/APNs errors; a permanent `Unregistered`/410 → `unregistered` result and the bench **disables that device token on first occurrence** (post-verification; not after 5 strikes — permanent means permanent). Bench: records `last_push_status/last_push_at`; relay unreachable → RQ retry with backoff, then log + drop (push is an accelerator; DB catch-up is the guarantee). App calls `ack_push(push_id)` on notification-open → end-to-end delivery-rate metric. Weekly cron (added to `jarvis/hooks.py:174-226`) purges tokens of devices unseen >90 days.

## 3. Mobile API contract

**Reused as-is** (Bearer works on every whitelisted endpoint today via `frappe/auth.py:629-644,647-684`; cookie-less bearer requests short-circuit CSRF, `frappe/auth.py:81-97`) — `jarvis.chat.api`: `send_message` (`chat/api.py:381`; returns `{ok, run_id, message_id, conversation_id}` `:564-567`), `list_conversations` (:62), `search_conversations` (:90, `{rows,total,has_more,start,page_length}` envelope), `create_conversation` (:300), `create_or_focus_empty` (:136), `get_conversation` (:164), `retry_message` (:769), `get_canvas` (:204), `preview_file` (:275), `get_chat_ui_settings` (:570; carries `time_zone` :597), `get_usage` (:649), `set_conversation_model` (:690), `set_conversation_thinking` (:745), `archive_conversation` (:313), `rename_conversation` (:347), `set_star` (:360), `list_tools` (:52), `warm_session` (:732); stock `upload_file`. `jarvis.chat.approvals_api`: `list_approvals_page` (:138), `get_approval` (:233), `pending_count` (:273), `decide` (:289). `jarvis.chat.actions_api`: `confirm_tool` (:235), `list_pending_confirmations` (:316), `apply_action` (:161), `get_doctype_form_meta`, `load_doc`.

**Reused for web-parity features (scope addendum):** `jarvis.chat.custom_skills_api.*` (skill list/get/create/update + `apply_custom_skills` — the container skills push; see §6 gate), `jarvis.chat.macros_api.*` (list/get/create/update/run — run progress arrives as `macro:progress`/`macro:done` events already in the §4 catalog), `jarvis.chat.agents_api.*` (gallery + detail + start-chat), `jarvis.chat.filebox.*` (list/upload/preview/`drop_file` send-to-chat), `jarvis.chat.learned_api.*` (learned-suggestion review cards), `jarvis.chat.api.set_auto_apply` (:603 — consequential toggle; see §6 gate), `jarvis.account.get_account` (plan/subscription/renewal for the account screen; billing actions stay web — the app links out, no in-app purchase flow), and `frappe.desk.search.search_link` (composer @-mentions, same as the SPA). All are existing whitelisted endpoints — zero new server code for these features; the mobile work is client-side plus allowlist entries.

`sync_conversation` is the cursor primitive: same field list as `get_conversation` (:173-189) with `WHERE seq > %(after_seq)s ORDER BY seq` (seq is monotonic per conversation, `:821-827` — clock-skew immune). The client additionally dedupes by message `name` (a rare `_next_seq` race can duplicate a seq — audit §15; dedupe makes the cursor safe regardless). `pending_confirm.list_for_owner` is folded in (`pending_confirm.py:157-197`) — one round-trip on foreground.

**`active_run` plumbing (post-verification — new, required).** No run identifier is persisted anywhere today (managed mode holds it only in worker memory). Change: the worker stamps `Jarvis Conversation.active_run_id` (+ `active_run_started_at`) at `run:start` and clears both at `run:end`/`run:error`; `sync_conversation` returns it; `stop_run` validates against it.

**`stop_run` (post-verification — bound).** Sets `jarvis:cancel:<conversation>:<run_id>` in Redis (TTL 720 s matching `_AGENT_TURN_WORKER_TIMEOUT`, `chat/api.py:37`) — but only after verifying, in SQL with `WHERE owner=%s`, that the caller owns `conversation` **and** that `active_run_id == run_id`. The worker's event-streaming loop in `turn_handler.py` checks the flag per event batch, closes the openclaw WS, finalizes the placeholder as `error="Stopped by user"`, publishes `run:error(stopped)`. Cooperative only; a late stop returns `{ok:false, code:"run_finished"}`. Cross-user cancellation is structurally impossible (flag key includes the owner-checked conversation).

**Versioning for a shipped binary.** (a) The mobile surface is namespaced `jarvis.mobile.*` — SPA refactors can't break it; (b) **additive-only** evolution everywhere: fields never removed/renamed/repurposed, new behavior behind optional params, unknown response fields ignored by the client; (c) breaking change ⇒ new method name (`sync_conversation_v2`), old kept ≥6 months; (d) app sends `X-Jarvis-App-Version`; `app_config.min_app_version` drives a blocking force-upgrade screen — the server-side kill switch (server also refuses below-floor versions with `error.code="upgrade_required"`); (e) push schema versioned by `data.v`; (f) JS-level fixes via EAS Update; a shared TS package pins request/response schemas. No `/v2` URL scheme, no BFF.

## 4. Realtime contract

**Verified: bearer auth works today — no server gap.** The socket.io auth middleware accepts cookie **or** `Authorization` header (`realtime/middlewares/authenticate.js:26-43`), forwards it to `/api/method/frappe.realtime.get_user_info` (`:52-55, 67-68`), which explicitly handles bearer sessions (`frappe/realtime.py:142-156`). Two constraints the middleware imposes on the RN client (backend contract, documented for the client team):
1. **Origin must match host** (`authenticate.js:21-24`) and RN sends no Origin → connect with `transports:['websocket']`, `extraHeaders: {Origin: "https://<site>", Authorization: "Bearer …", "X-Frappe-Site-Name": "<site>"}` (site-name resolution honors that header, `authenticate.js:92-93`; note the Origin value also serves as the node server's base URL for the auth fetch — it must be the real site origin, not a placeholder).
2. **Namespace = site hostname** (wildcard namespaces `realtime/index.js:23`, validated `authenticate.js:14-19`).

On connect the socket auto-joins `user:<user>` (`realtime/handlers.js:5`; room name `frappe/realtime.py:167-168`) — exactly where `publish_to_user` sends `jarvis:event` (`jarvis/chat/events.py:19,65-67`). Zero subscription code. Auth happens once per connect: the client refreshes a near-expiry token *before* connecting (foreground only, §1); the token is supplied via a function so reconnects pick up rotated tokens. If Expo's WS layer ever strips `Origin`, the fallback is a one-line middleware allowance for header-authenticated sockets — flagged, not assumed.

**Event catalog** (single channel `jarvis:event`): `run:start` (`turn_handler.py:430-431`), `assistant` (cumulative `text` + `delta` — idempotent to re-apply, safe to drop frames; `events.py:55-60`), `tool` start/end with `tool_title` (`events.py:35-53`), `canvas`, `run:end` (:780-781), `run:error` (:512, :756, :1187; `stale_scan.py:72`), `action:pending` (`jarvis/api.py:682-690`), `conversation:new` (`proactive.py:58-61`), `conversation:renamed`, `macro:progress/done/merged` (`macros.py:320,356,205`).

**Reconnect/catch-up.** Socket = foreground-only accelerator (locked); no server-side replay exists and none is added. On connect / reconnect / app-foreground / push-tap: `home_snapshot` + `sync_conversation(open_conv, max_seq)`, debounced 2 s; refetch the open conversation on `run:end` (mirrors the SPA's `onResync`). Cumulative `assistant.text` + seq cursor make stream loss recoverable by construction.

## 6. Security

**QR/pairing token properties:** 256-bit CSPRNG; 120 s TTL; single-use (atomic GETDEL, the proven `pending_confirm.py:141-142` pattern); stored hashed; bound to creating user + web session + purpose; exchanged for a second 60 s single-use `browser_code` so **no secret ever appears in a URL, browser history, or access log**; **2-digit number-matching confirmed on the web side** (anti shoulder-surf / reverse-phishing); consent screen retained; live "paired — revoke" feedback closes the remaining social window; every claim logged with IP/UA.

**JRV-SEC-005 (plaintext Password `db_set`, audit §14):** zero new secrets use `db_set`; pairing secrets and device-token mappings are hashed; `push_relay_key` uses `set_encrypted_password`; push tokens are ORM-written Password fields. **Piggyback:** the PR touching `Jarvis Settings` converts `write_connection`'s `db_set` writes (`jarvis/onboarding.py`, audit §12) to `set_encrypted_password`.

**IDOR (JRV-SEC-002, audit §14):** mobile reuses the 7 unguarded endpoints (`get_conversation`, `get_canvas`, `archive`, `rename`, `set_star`, `send_message`, `retry_message`) — bearer tokens widen their reachable surface, so **landing the owner-check fix is a hard mobile-GA prerequisite**, together with owner checks on `set_conversation_model`/`set_conversation_thinking` (verified unguarded: `frappe.db.exists` only, `chat/api.py:703, :753`). Every *new* endpoint copies the explicit `WHERE owner=%s` pattern (`chat/api.py:75-86`; `approvals_api.py:159-163`), never bare `get_doc`.

**Scope-`all` containment (post-verification; updated by the scope addendum).** A Frappe bearer token carries the user's full API power — scopes are client-level only (`frappe/oauth.py:227-235`). The same `enforce_mobile_token_policy` auth hook therefore enforces a **method allowlist for mobile-client bearer sessions**: `jarvis.chat.api.*`, `jarvis.chat.approvals_api.*`, `jarvis.chat.actions_api.*`, `jarvis.chat.custom_skills_api.*`, `jarvis.chat.macros_api.*`, `jarvis.chat.agents_api.*`, `jarvis.chat.filebox.*`, `jarvis.chat.learned_api.*`, `jarvis.mobile.*`, `jarvis.account.get_account`, `upload_file`, `frappe.desk.search.search_link`, `frappe.integrations.oauth2.*`, `frappe.realtime.get_user_info`. Mobile tokens still cannot reach desk/admin or any other endpoint. The allowlist is config-extendable per site.

**Parity-driven gate expansion (scope addendum).** Two endpoints the original design excluded are now allowlisted for web parity, and both are audit-flagged as under-gated server-side (SEC-B family): `apply_custom_skills` (any session user can trigger a container skills push/restart — audit §9) and `set_auto_apply` (flips the site-wide write-confirmation guardrail with no `only_for` gate). Including them on mobile does not *create* these holes — the web SPA already exposes both — but bearer tokens widen their reach, so **their server-side gating fixes join JRV-SEC-002/SEC-B owner checks in the hard mobile-GA gate**: `set_auto_apply` gains a permission check + an audit-log entry; `apply_custom_skills` gains owner/role gating + rate limiting before mobile GA.

**No write-authorizing material in push:** confirm tokens stay exclusively in the realtime channel + `list_pending_confirmations` (both owner-bound); `confirm_tool` remains the sole execution path (owner+conversation+single-use, `actions_api.py:283-286`).

**Relay abuse prevention:** template-based API (no tenant-supplied free text), per-tenant hashed bearer keys with admin rotation; per-tenant rate limit (60/min sustained, burst 120) **fail-closed**; 4 KB payload cap; template/category allowlist; no payload logging; `127.0.0.1` + non-root + TLS (not SEC-E's posture). A compromised tenant key can spam only Jarvis-shaped notifications to tokens it already holds; a compromised relay yields signing creds but no device tokens, content, or credentials usable against any bench.

**OAuth provider hardening:** guest dynamic client registration **disabled** per tenant site (ships enabled by default — `oauth2.py:335-344`); `skip_authorization=0` pinned + monitored (a site-level toggle can otherwise silently remove consent, `oauth2.py:118`); PKCE-not-mandated documented as an accepted limitation with the compensations in §1.

**Bench rate limits (Redis counters, fail-closed — explicitly not the audit's SEC-D fail-open pattern):** `pairing.claim` 5/min/IP + 20/h/site; `create_pairing_session` 3/10 min/user; `web_login` 5/min/IP; `register_device` 10/h/user; `stop_run` 30/h/user.

**Threat model summary:** hostile network → TLS+PKCE, no client secret; stolen phone → Keychain/Keystore (`AFTER_FIRST_UNLOCK`) + per-device revoke + whitelist hook + 1 h TTL; QR hijack → TTL/one-time/hash/number-matching/web-visibility (+consent as friction); stolen refresh chain → whitelist hook + new-device alert + everywhere-signout; push-channel snooping → generic template payloads, preview opt-in, no tokens; relay compromise → creds only, one-way trust; relay-key leak → templates + quotas cap blast radius; replay → single-use everywhere (pairing secret, match confirmation, browser code, confirm tokens, conditional approval flip); malicious tenant → relay quotas + key scoping; cross-user cancel → conversation-bound stop flag; token-DB dump → hashes, not tokens; scope escalation → method allowlist.

## 7. Failure modes & edge cases

- **Token expiry mid-task:** the run is worker-owned and unaffected; app refreshes on next foreground 401, retries once, catches up via `sync_conversation`. Refresh rejected (revoked/non-2xx) → wipe secure-store → pairing screen; device row survives for re-pair.
- **Push not delivered:** never load-bearing — foreground `home_snapshot` + seq catch-up reconcile; badge from `pending_count` (`approvals_api.py:273-286`) rides the push payload server-side; persistent failure visibly disables the device (`last_push_status`).
- **Approval decided elsewhere:** losing writer gets "decided concurrently" (`approvals_api.py:314-318`); consumed confirm token → `InvalidConfirmation` (`actions_api.py:226-232, 285-286`). App renders both as state ("Already handled") + refetch; other devices get the visible collapse-id replacement (§2).
- **Multi-device consistency:** events are user-room scoped (`events.py:65-67`) → all connected devices converge; pushes fan out to all enabled devices; seq-cursor sync is idempotent (+ name-dedupe); DB arbitrates.
- **Tenant site unreachable/suspended:** cached reads + offline banner, compose disabled, backoff; no pushes can originate (they come from the tenant) — honest degradation. Destroy → relay deregistration (§8).
- **App killed mid-stream:** zero loss — messages seq-persisted; streaming placeholders resolved by `stale_scan`/`turn_recovery` crons (`hooks.py:176-187`); reopen → cursor fetch; completion push still lands. Locked phone: token readable after first unlock (§1), so push-tap → sheet works from the lock screen.
- **Confirm token expired (900 s) in pocket:** approval sheet shows expiry gracefully → deep link opens conversation; agent re-parks next turn.
- **`stop_run` races `run:end`:** TTL'd flag, worker-checked; late stop = benign `{cancelled:false}`.
- **Clock skew:** all TTLs server-enforced (`frappe/oauth.py:230`); cursors are `seq`; `home_snapshot` echoes server-issued `updated_since`; client uses relative `expires_in`, never wall-clock comparisons; display timezone from `get_chat_ui_settings.time_zone` (`chat/api.py:597`).

## 8. Ops & rollout

**Relay ops:** own repo `jarvis-push-relay`; systemd (dedicated user, `127.0.0.1` behind nginx TLS); TOML/env config (FCM SA json, APNs p8 — 0600 files, not doctype fields); notification templates versioned in the repo (reviewable, rollbackable — not Desk-editable); Prometheus metrics (sent/failed by platform + error class, per-tenant rates, ack-derived delivery rate); structured logs without content; alerts on APNs/FCM auth errors, tenant success-rate <95%/15 min, volume anomalies, relay down. v1 HA = single instance — safe because benches retry from RQ and push is never load-bearing. Admin Fleet Console gets a relay-status card + per-tenant key rotation.

**Tenant lifecycle:** provision → admin registers tenant at relay + delivers key via `get_connection`; suspend → relay registration disabled (benches get 403, back off); destroy → relay deregistration wired into the existing `fleet/lifecycle.py` destroy path. No orphaned push capability survives teardown.

**Staged rollout (3–4 months, 2–3 devs):**
1. **Weeks 1–3 — demo slice:** OAuth Client fixture + `app_config` + bearer reuse of chat endpoints + socket `extraHeaders` auth + `sync_conversation`. *Demo: in-app browser login, chat with streaming + catch-up.*
2. **Weeks 4–7:** full QR pairing (incl. number-matching) + device registry + auth hook (whitelist + method allowlist) + relay MVP + `run:end`/`run:error` pushes. *The demo moment: leave the office, get "task finished."* Internal TestFlight/Play-internal.
3. **Weeks 8–11:** approval/confirm pushes + in-app approval sheet contract + `stop_run` + `active_run_id` plumbing + device management/revocation UX + notification prefs + rate limits. **Parity surfaces land here too** (skills, macros incl. live run progress, agents, files, account/settings screens) — server-side this is only allowlist entries + contract tests, since all endpoints exist; the effort is client-side.
4. **Weeks 12–16:** hardening (pairing pen-test, relay load test, fail-closed verification, GA-gate fixes confirmed landed — JRV-SEC-002/SEC-B owner checks **plus** the `set_auto_apply`/`apply_custom_skills` gating from §6, Time-Sensitive entitlement review), pilot tenants behind `Jarvis Customer.mobile_enabled`, staged store rollout. `min_app_version` kill switch live from day one. Note: full parity in one v1 is ambitious for 2–3 devs in 16 weeks — if the schedule slips, the drawer ships with skills/macros/agents/files behind a server-side feature flag (`app_config.features`) so the binary doesn't have to wait.
